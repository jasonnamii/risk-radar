# 리스크 대응 전략과 설계

## 리스크 선호도 프레임워크 (Risk Appetite Framework)

### 개념 구분

**리스크 선호도** (Risk Appetite)
- 조직이 "전략적으로" 감당할 의향이 있는 리스크의 상한
- CEO와 이사회가 설정
- 예: "연 매출 대비 5% 손실까지는 감수"

**리스크 허용도** (Risk Tolerance)
- 각 운영 부문에서의 구체적인 가드레일
- Risk Appetite를 각 부서에 cascade
- 예: 영업팀은 "신용위험 3%", 개발팀은 "가용시간 99.9%"

### 선호도 정의 프로세스

**Step 1: 전략 목표 확인**
```
"우리는 3년 내 매출 2배를 목표로 한다"

이를 위해 수용 가능한 리스크는?
  - 시장 변화 대응 속도 중시 → 기술 리스크 높음 용인
  - 규정 위반은 절대 금지 → Compliance 리스크는 제로
  - 신규 사업 진출 → 전략 리스크는 높음 용인
```

**Step 2: 리스크 타입별 선호도 수치**

| 리스크 유형 | 정의 | 예시 | 선호도 수준 |
|---|---|---|---|
| 예방가능 리스크 | "발생하면 안 되는 것" | 부정행위, 사고, 규정 위반 | Near-zero (0%) |
| 전략 리스크 | "경영진 의사결정의 결과" | 신규 시장 진출, M&A | 중~높음 (20~40%) |
| 외부 리스크 | "통제 불가능한 것" | 경기 침체, 경쟁사 혁신 | 중~낮음 (5~15%) |

**Step 3: KRI (Key Risk Indicator) 설정**

```
전략 목표: 신규 시장 1곳 진출 (높은 전략 리스크 용인)

KRI 설정:
  ① 신규 시장 실패율: 30% 이상 (경고)
  ② 초기 18개월 손실: 매출의 20% (경고)
  ③ 핵심인력 이탈: 20% 이상 (경고)
  
Threshold:
  ① 3건 중 1건 이상 실패 → 전략 재검토
  ② 초기 손실 20%+를 초과하면 → 즉시 재무 강화
  ③ 인력 20% 이탈 → 보상/조직 문화 재점검
```

### 선호도 커뮤니케이션

조직 전체에 cascade되어야 함:

```
CEO/이사회 설정:
  "전략 리스크는 20% 수준 감수. 규정 위반은 제로"
  
경영진 해석:
  "신사업에는 과감하게. 법적 위험은 극도로 보수적"
  
각 부문 실행:
  - 영업: 신규 고객 개척에 리소스 투입, 거래선 검증 강화
  - 법무: 모든 계약 사전 검토, 규정 위반 제로 정책
  - R&D: 신기술 실험 지원, 실패 용인 문화
```

## 리스크 대응 전략 매트릭스

### 4가지 기본 전략

#### 1. 회피 (Avoidance)

**정의**: 리스크를 야기하는 활동 자체를 하지 않음

**적용 기준**:
- 비용: 회피 비용 < 리스크의 EMV (기댓값)
- 대안: 리스크를 피하고도 목표를 달성할 다른 방법 존재
- 전략성: 우리의 핵심 전략이 아님

**예시**:
- "이 공급처의 신용도가 너무 낮다" → 공급처 변경 (회피)
- "이 기술은 아직 검증 부족" → 검증된 기술 사용 (회피)
- "이 시장은 규제 위험이 크다" → 시장 진출 포기 (회피)

**비용·난이도·효과**:
| | 비용 | 난이도 | 효과 |
|---|---|---|---|
| 비용 | 가능하면 낮음 | 쉬움 | 100% (리스크 제거) |
| 난이도 | 가능하면 높음 | 어려움 | 실제로는 원래 목표 못 달성 |
| 효과 | 명확함 | 명확함 | 리스크 완전 제거 |

**주의**: "회피"가 항상 현명한 것은 아님. 전략적 고민 필수.
- 예: "모든 리스크를 피하면 성장하지 못함"

---

#### 2. 전가 (Transfer)

**정의**: 리스크를 다른 주체에게 이전. 통상 계약·보험을 통해.

**적용 기준**:
- 대상: 리스크를 관리할 전문성 또는 규모의 경제 있는 주체
- 비용: 전가 비용(보험료, 계약 조건) < 리스크 임팩트
- 통제력: 우리가 충분히 통제·감시할 수 있어야 함

**예시**:
- 화재 보험: 건물 화재 리스크 → 보험사
- 공급 계약: 납기 지연 리스크 → 공급사 (계약 조항)
- 역할 아웃소싱: 개발 능력 부족 → 외부 개발사
- 지분 투자자: 자금 부족 리스크 → 투자자 (자본금 조달)

**비용·난이도·효과**:
| | 비용 | 난이도 | 효과 |
|---|---|---|---|
| 화재 보험 | 낮음 | 매우 쉬움 | 70~90% (전가되지 않은 부분 있음) |
| 아웃소싱 | 높음 | 매우 어려움 (새로운 리스크: 협력사 관리) | 50~80% |
| 계약 조항 | 낮음 | 쉬움 | 60~80% (협력사가 정말 이행하는가?) |

**주의**: 전가했다고 끝이 아님. 상대방이 실제로 관리하는지 감시 필수.
- 예: "보험은 들었는데 실제 사고 시 보험사가 거부" → 사전 확인 필수

---

#### 3. 완화 (Mitigation)

**정의**: 리스크의 확률 또는 임팩트를 감소시키는 조치

**가장 일반적인 대응.** 회피나 전가가 불가능할 때.

**확률 감소 vs 임팩트 감소**:

**확률 감소 예시**:
- 리스크: "일정 지연"
  - 조치: 버퍼 일정 추가, 리소스 증원, 리스크 조기 탐지 프로세스
  - 효과: 지연 확률 70% → 40%로 감소

- 리스크: "주요 인력 이탈"
  - 조치: 보상 인상, 경력 개발, 긍정 문화
  - 효과: 이탈률 20% → 10% 감소

**임팩트 감소 예시**:
- 리스크: "데이터센터 정전"
  - 조치: 중복 데이터센터 (지역 분산), 자동 failover
  - 효과: 정전 발생 시 영향 범위 100% → 10%로 감소

- 리스크: "신규 제품 실패"
  - 조치: MVP 먼저 테스트, 고객 feedback 빠르게 수집
  - 효과: 전체 손실을 1000만원 → 100만원으로 제한

**비용·난이도·효과**:
매우 가변적. 상황에 따라 다름.

---

#### 4. 수용 (Acceptance)

**정의**: 리스크 발생 시 감수할 것을 사전에 인정. 추가 조치 하지 않음.

**적용 기준**:
- 확률과 임팩트가 모두 낮음 (Green, Yellow 리스크)
- 완화·전가·회피 비용이 리스크 자체보다 큼
- 리스크가 전략적으로 허용 범위 내

**예시**:
- "동전 던지기 결과에 따라 회의 내용 결정" ← 확률 50%, 임팩트 무시할 수준 → 수용
- "PC 하드드라이브 고장" ← 확률 낮음, 영향 미미 (백업 있음) → 수용
- "경기 침체" ← 완화 불가능, 대응만 준비 → 수용

**"모니터링만 한다"의 의미**:
수용한다는 것이 "아무것도 안 한다"는 뜻은 아님.
- 여전히 리스크를 모니터링 (신호 포착)
- 발생 시 빠르게 대응할 준비 (contingency plan)

**비용·난이도·효과**:
| | 비용 | 난이도 | 효과 |
|---|---|---|---|
| 사전 비용 | 0 | 매우 쉬움 | 0 (발생 시 비용) |
| 실제 임팩트 | 리스크 전액 | 어려움 | 계획되지 않은 영향 |

---

### 대응 선택 의사결정 트리

```
Step 1: 이 리스크를 완전히 피할 수 있는가?
  YES → 회피 검토 (전략적 가치 vs 리스크)
  NO → Step 2

Step 2: 이 리스크를 전문가/보험에 이전할 수 있는가?
  YES → 전가 검토 (전가 비용 vs 리스크 임팩트)
  NO → Step 3

Step 3: 이 리스크의 확률이나 임팩트를 줄일 수 있는가?
  YES → 완화 전략 설계 (비용 대비 효과)
  NO → Step 4

Step 4: 이 리스크를 감수할 것인가?
  YES → 수용 (모니터링 + Contingency plan)
  NO → 우려 상황. 리스크 선호도 재검토 필요
```

## 고급: Antifragility 설계

### 개념 (Taleb)

Robust (견딤) ← → Antifragile (성장)

리스크를 "제거"하기보다 "이용"하는 시스템 설계.

### 설계 원칙

#### 1. 옵션성 (Optionality)

**정의**: "차후 의사결정을 유보할 권리"

**예시**:
- MVP 출시: 전체 기능을 다 만들지 말고, 코어만 먼저
  - 옵션: 시장 신호에 따라 방향 조정
  - vs 전통: 모든 것을 미리 정해서 대규모 개발
  
- 공급처 다변화:
  - 옵션: 한 공급처에 문제 시 다른 곳으로 빠르게 전환
  - vs 전통: "최저가 공급처 1개"로 고정

#### 2. 비대칭성 (Asymmetry)

**정의**: 손실은 제한적, 수익은 무한.

**Barbell Strategy**:
```
포트폴리오 구성:
  90% 극도로 안전한 자산 (국채, 현금)
    → 손실 제한적 (최대 0)
  10% 극도로 고위험 고수익 (스타트업 투자, Option)
    → 수익 무한 (극단적 성공 시 10배 이상)

평균 포트폴리오 (중간 위험):
  50% 중간 위험 자산
    → 손실 크고, 수익도 제한적 (최악)
```

**비즈니스 예시**:
- 제품 포트폴리오: "80% 안정적 기존 제품 + 20% 신기술 실험"
- 투자: "90% 검증된 기술 + 10% 미래 기술 R&D"
- 팀: "코어팀은 안정적 보상, 신사업 팀은 고수익 포텐셜"

#### 3. 중복성 (Redundancy)

**정의**: 한 부분 실패 시 다른 부분이 커버

**예시**:
- 데이터센터: 1곳 → 3곳 (지역 분산)
- 공급처: 1개 → 3개 (가격은 상승하나, 리스크 감소)
- 핵심 인력: 1명 독점 → 2명 이상 교차훈련

**"낭비"처럼 보이지만 리스크 컨트롤 가치**:
- 추가 비용 10% 증가
- 가용시간 99% → 99.9%로 개선 (시간당 수익 100배 사이트라면, 가용시간 0.9% 개선 = 매년 천만원대 가치)

## HRO (High-Reliability Organizations) 대응 원칙 체크리스트

Weick의 5가지 원칙을 대응 설계에 적용:

| 원칙 | 대응 설계에의 의미 |
|------|---|
| 실패 집착 | 리스크 Red = "이미 실패"라고 가정. 당연히 발생할 것처럼 대응 설계 |
| 단순화 저항 | "이 대응안이 항상 통할까?" 재문. 예외 케이스 고려 |
| 운영 민감성 | 대응안이 종이에만 있지 말고, 정기적 훈련·시뮬레이션 실행 |
| 회복력 | "사고 예방"이 아니라 "사고 후 빠른 회복" 목표. Contingency 자세 |
| 전문성 존경 | 대응 설계 시 "높은 직급"보다 "그 분야 전문가" 의견 우선 |

```
구체적 조치:
  1. [첫 번째 조치]
    - 담당자: [누가]
    - 시작일: [언제]
    - 완료일: [언제]
    - 비용: [얼마]
    - 효과: 확률 [X]% → [Y]% 감소 / 임팩트 [A]→[B]

  2. [두 번째 조치] ...

대응 후 P×I: [변경된 확률] × [변경된 임팩트] = [새로운 등급]

```
