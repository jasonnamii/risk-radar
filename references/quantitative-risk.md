# 정량적 리스크 분석

## Monte Carlo Simulation

### 개념

많은 수의 난수 표본을 이용하여 확률 분포를 시뮬레이션. "가장 가능성 높은 값"이 아니라 "가능한 결과들의 전체 분포"를 도출.

### 프로세스

**Step 1: 입력 변수의 분포 정의**

```
예: 소프트웨어 프로젝트 일정 추정

변수 1: 요구사항 분석 기간
  분포: 정규분포, 평균 4주, 표준편차 1주
  
변수 2: 개발 기간
  분포: 정규분포, 평균 12주, 표준편차 3주
  (높은 불확실성)
  
변수 3: 테스트 및 버그 수정 기간
  분포: 정규분포, 평균 3주, 표준편차 1.5주
```

**Step 2: 상관관계 정의**

- 변수들이 완전히 독립적인가, 상관관계가 있는가?
- 예: "개발 기간이 길면 테스트 기간도 길다" (양의 상관)
- 상관계수 -1~1로 표현

**Step 3: 수천 번 시뮬레이션**

각 시뮬레이션마다:
1. 각 변수에서 난수 하나씩 추출 (정의된 분포에서)
2. 변수들을 합산: 총 프로젝트 기간 = 분석 + 개발 + 테스트
3. 결과 기록

1000~10000번 반복 → 분포 형성

**Step 4: 결과 분석**

```
결과 분포:
  평균 (기댓값): 19주
  표준편차: 2.8주
  5th percentile (낙관): 14주 → 95% 확률로 이보다 오래 걸림
  50th percentile (중위): 19주
  95th percentile (비관): 25주 → 95% 확률로 이 이내에 끝남
```

### 강점

- 상관관계 포착: 변수들의 상호작용 반영
- 비선형 효과: 복잡한 공식도 시뮬레이션으로 처리
- 전체 분포: "평균"뿐 아니라 "극단값" 이해 가능

### 약점

- 입력 분포 추정이 어려움: 데이터 부족 시 임의성 높음
- "garbage in, garbage out": 입력이 잘못되면 결과도 의미 없음
- 설명의 복잡성: 경영진에게 설득하기 어려움

### 비즈니스 적용

- 프로젝트 일정·원가 추정
- 재무 모델링 (매출, 비용, 이익)
- 포트폴리오 최적화 (여러 프로젝트의 동시 실행)

## Fat Tails와 Power Laws (멱함수 분포)

### 정규분포 vs Fat-tail 분포

**정규분포 (Normal Distribution)**
- 종 모양
- 극단값이 매우 드뭄: 5-sigma (5 표준편차) = 1/3.5M
- 현실: "사건이 절대 일어나지 않음"
- 예: 키, 몸무게, 측정 오차

**Fat-tail 분포 (Power-law)**
- 꼬리가 두꺼움 (극단값이 더 자주 발생)
- 5-sigma = 1/10K (훨씬 자주 일어남)
- 특징: "큰 사건이 전체 결과를 지배"
- 예: 부의 분포, 기업 규모, 자연재해, 팬데믹 피해

### Pareto 원칙 (80/20 법칙)

**관찰**: 많은 현상에서 전체 결과의 80%가 20%의 원인에서 나옴

- 전 세계 부의 80% = 상위 20% 인구
- 기업 매출의 80% = 상위 20% 고객
- 웹사이트 트래픽의 80% = 상위 20% 페이지
- 산업 재해의 80% = 상위 20%의 위험 요소

**의미**: 평균을 기준으로 계획하면 극단값 대비 무방비.

### 비즈니스에서의 Fat-tail 리스크

**예 1: 금융 시스템**
- 일상적 변동: 정규분포 가정
- 2008년 금융위기: Fat-tail 사건 (5-sigma 넘음)
- 통상적 리스크 모델은 이를 "거의 불가능"으로 평가 → 준비 X → 재난

**예 2: 기술 산업**
- 대다수의 스타트업: 실패
- 소수의 스타트업: 극도의 성공 (Facebook, Google, Airbnb)
- 평균만 본다면 "모두 실패할 것" 예상 → 틀림

**예 3: 재해·감염병**
- Fort McMurray 대화재 (2016): 캐나다 화재 피해의 70% = 단일 사건
- COVID-19: 예측할 수 없는 Global Disruption
- 일상적 패턴 분석 = 쓸모없음

### 대응 전략

**1. 불가능을 버리기**
- "5-sigma는 불가능하니까 무시" ✗
- "Fat-tail에서는 5-sigma가 1/10K 정도 되니까 시뮬레이션 필수" ✓

**2. Taleb의 Barbell Strategy**
- 90% 극도로 안전한 자산 + 10% 고위험 고수익
- 손실은 제한적 (90%의 안정성)
- 수익은 무한 (10%에서의 극단적 성공)

**3. 옵션 사고**
- "정확히 예측" 포기
- 대신 "여러 선택지를 유보" (옵션 유지)
- 미래가 명확해지면 그때 큰 베팅

### 팻테일 검출

**신호**:
- "이게 일어나면 결국 나머지를 좌우한다" → Fat-tail
- "이런 극단적 시나리오가 주기적으로 발생" (1년에 한 번 등) → Fat-tail

## 리스크 속도 (Risk Velocity)

### 정의

리스크가 "위협"에서 "현실"로 얼마나 빠르게 구체화되는가?

### 영향 요소

| 요소 | 빠른 속도 | 느린 속도 |
|------|---------|---------|
| 산업 역학 | 변화 극심 (AI, 생명공학) | 변화 완만 (제지, 석탄) |
| 위협 본질 | 급작스러운 (기술 혁신, 사고) | 점진적 (규제 변화, 인구) |
| 지리적 집중 | 한곳에 집중 (본사 의존) | 분산 (글로벌 다중 사이트) |
| 기술 발전 속도 | 지수적 (소프트웨어) | 선형 (기계) |
| 조직 대응 속도 | 느림 (대규모, 경직) | 빠름 (소규모, 민첩) |

### 예시

**높은 속도 (1~6개월)**
- "경쟁자가 우리 주력 상품을 대체하는 기술 출시" → 매출 급락
- "핵심 엔지니어 이탈" → 프로젝트 지연
- "보안 사고" → 평판 즉시 손상

**낮은 속도 (1~5년)**
- "새로운 규제 진행 중" → 아직 영향 X
- "인구 고령화" → 장기 트렌드
- "기술 standard 변화" → 점진적 영향

### 의사결정 영향

속도가 높으면:
- "관찰과 대응" 스타일 = 늦음 ✗
- "사전 예방" 필수 ✓

속도가 낮으면:
- "미리 다 준비" = 환경 변화에 맞지 않음 ✗
- "지속적 모니터링 + 적응" 필수 ✓

## 연쇄 실패 (Cascade Failure)

### 정의

시스템의 일부 실패 → 그 부분의 부하가 이웃으로 이전 → 연쇄적 붕괴

### 상호의존 유형

**물리적 의존** (Physical Interdependency)
- 한 부분이 물리적으로 다른 부분에 입력 공급
- 예: 전력망 (발전소 X → 변전소 X → 가정 정전)

**사이버 의존** (Cyber Interdependency)
- 정보 흐름 기반
- 예: 클라우드 서비스 (데이터센터 장애 → API 불가 → 모든 앱 다운)

**지리적 의존** (Geographic Interdependency)
- 같은 지역의 공유 자원
- 예: 지진 (송전탑 X → 통신 X → 병원 X)

### 실패 경로

```
1단계: 초기 실패 (어떤 부분이 먼저 깨짐)
  예: 소프트웨어 배그

2단계: 부하 전이 (그 부분의 역할이 이웃으로)
  예: 다른 서버가 그 부분의 요청 처리
  → 그 서버 과부하 시작

3단계: Threshold 초과 (이웃도 견딜 수 없음)
  예: 과부하 서버가 응답 불가

4단계: 연쇄 붕괴 (다음 다음으로 전파)
  예: 다른 서버들도 순차적 다운
```

### 예시

**2003년 미국 대정전**
- 초기: 오하이오의 한 변압기 고장
- 부하 전이: 이웃 변압기들이 그 부하 흡수
- 연쇄: 50분 내 Northeast 전역 정전 (5000만 명)

**2024년 글로벌 IT 장애 (CrowdStrike)**
- 초기: CrowdStrike 소프트웨어 버그
- 부하 전이: 의존하는 모든 컴퓨터 다운
- 연쇄: 항공사, 병원, 금융사 일괄 마비

### 방지 전략

1. **결합도 감소**: 부분들 간 독립성 최대화
   - 예: 마이크로서비스 아키텍처
   
2. **버퍼 추가**: 부분의 용량을 peak 대비 여유있게
   - 예: 서버 자원 50% 여유 (peak 150% 견딜 수 있도록)
   
3. **Circuit Breaker**: 한 부분의 실패가 다음으로 전파되는 것을 차단
   - 예: API 호출 실패 시 즉시 요청 거절 (다른 서버까지 기다리지 않음)
   
4. **모니터링**: 초기 신호 포착 후 빠른 격리
   - 예: 한 서버 다운 감지 → 즉시 트래픽 우회

## Expected Value (기댓값)와 의사결정

### EMV 계산

```
Expected Monetary Value = Σ(Probability × Impact)

예: 투자 결정
  시나리오 A: 60% 확률로 +100만원
  시나리오 B: 40% 확률로 -50만원
  
  EMV = 0.6 × 100 + 0.4 × (-50)
      = 60 - 20
      = +40만원 (기댓값 양수 → 투자 권고)
```

### 강점

- 단순 명확: 한 숫자로 의사결정 가능
- 수학적 정당성: 정보 활용 최적화

### 약점

**1. 확률이 정확하지 않음**
- 어디서 나온 60%인가? 추측일 가능성 높음
- Fat-tail 리스크는 확률을 정할 수 없음

**2. 인간은 risk-neutral이 아님**
- Prospect Theory: 손실을 이득의 2배로 평가
- EMV가 양수라도, 손실 가능성이 크면 거부

**3. 하나의 극단값 언급 불가**
- 99%의 경우 +1원, 1%의 경우 -100억?
- EMV = -1,000만원 (부정적)
- 하지만 "거의 항상 이득"인데 EMV는 손실 제시

### 개선: 확률 + Monte Carlo + Scenario

```
EMV만: 기댓값 +40만원 → 의사결정 불충분

EMV + Monte Carlo:
  분포 형태: 정규분포? Fat-tail?
  극단값: 5% 확률로 손실 500만원?
  
EMV + Scenario:
  낙관: 70% 확률로 성공 (수익 200만원)
  기본: 20% 확률로 break-even (0원)
  비관: 10% 확률로 실패 (손실 100만원)
  → 우리는 10% 손실을 견딜 수 있는가?
```
