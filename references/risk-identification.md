# 리스크 식별 기법

## Pre-mortem (사전 검시법)

### 배경

Gary Klein의 연구. "사후 검시(post-mortem)"의 역 프로세스. 프로젝트 실패를 미리 상상하고 "왜 실패했는가"를 추론.

### 프로토콜

**Step 1: 장면 설정**
- 현재 날짜로부터 [프로젝트 완료 예정일] 후를 상상
- "프로젝트가 완전히 실패했습니다"라고 공언
- 모든 참여자가 이를 역사적 사실처럼 받아들임

**Step 2: 개별 작성 (10분, 침묵)**
- "이 프로젝트가 왜 실패했는가?"를 개별적으로 작성
- 가능한 실패 원인 목록화
- 비판 없이, 가능한 많이 작성

**Step 3: 그룹 공유 및 정렬**
- 각자의 실패 원인을 공유
- 유사한 것들끼리 묶음
- 반복되는 패턴 파악

**Step 4: 대응 설계**
- 각 실패 원인별로 "이를 예방하려면 어떻게?"
- 예방 조치 구체화

### 효과

**실증 연구**:
- 일반적 계획: 성공률 약 55%
- Pre-mortem 실행: 성공률 약 85% (30% 향상)
- 결론: Prospective Hindsight가 다른 인지 경로를 활성화

### 원리

**왜 효과적인가?**
- 정상적 사고: "우리가 성공하려면?" → 낙관 편향 (할 수 있는 것만 나열)
- Pre-mortem: "우리가 실패했다" → 실패의 인과를 추론 (숨겨진 위험 노출)
- 심리: 실패는 이미 일어난 것이므로 "현실직시" → 더 정직한 분석

### 비즈니스 적용

- 신규 프로젝트 착수 전: 필수
- 의사결정 전: "만약 이 결정이 재앙이 되었다면?"
- 팀 토론: 불안감 없이 부정적 시나리오 이야기 가능

## FMEA (Failure Mode & Effects Analysis)

### 정의

체계적으로 시스템의 모든 부분을 분석하여 어떤 방식으로 고장날 수 있는지, 그 고장이 전체에 미치는 영향을 평가.

### 5단계 프로세스

**Step 1: 시스템 정의**
- 분석 대상 제품/프로세스 경계선 설정
- "무엇을 분석할 것인가"를 명확히

**Step 2: 고장 모드 열거 (Failure Modes)**
- 각 부품/단계별로 "어떻게 고장날 수 있는가?" 나열
- 부품별 가능한 모든 고장 양식 도출

예 (소프트웨어 배포):
- 환경변수 입력 오류
- 테스트 미충분
- 코드 리뷰 누락
- 의존성 버전 충돌
- 롤백 실패

**Step 3: 영향 분석 (Effects)**
- 각 고장 모드가 시스템 전체에 미치는 영향
- "그렇게 되면 뭐가 깨지는가?"

**Step 4: 심각도·빈도·탐지 평가**

| 메트릭 | 정의 | 평가 |
|-------|------|------|
| 심각도 (S) | 고장의 영향이 얼마나 심각한가? | 1~10 점 |
| 빈도 (O) | 이 고장이 얼마나 자주 일어나는가? | 1~10 점 |
| 탐지 (D) | 이 고장을 얼마나 빨리 감지할 수 있는가? | 1~10 점 (높을수록 탐지 어려움) |
| RPN | Risk Priority Number = S × O × D | |

**Step 5: 대응 설계**
- RPN이 높은 항목부터 대응
- 설계 개선 또는 통제 강화

### 강점과 약점

**강점**:
- 체계적. 빠뜨린 위험이 적음
- 정량화. 우선순위 명확
- 구체적. 대응이 구체화됨

**약점**:
- RPN 게이밍: "이 항목은 빈도 1"이라고 낮게 평가 (데이터 부족)
- 시스템 상호작용 미포착: 여러 결함의 동시 발생 고려 X
- 시간 소모: 세밀한 분석에 리소스 많음

### 비즈니스 적용

- 제조: 제품 설계 단계에서 필수
- 소프트웨어: 아키텍처 리뷰 시
- 프로세스: 운영 위험 식별

## Bow-tie 분석 (활 모양 분석)

### 구조

```
            Causes (원인들)      Top Event      Consequences (결과들)
            
                  ↘             ↓              ↙
                    ◉━━━━━━━━━━━━━━━━━━━━━━━━━━━◉
                  ↗             ↑              ↖
            
           Barriers (좌측)   Central Event   Barriers (우측)
```

**좌측**: 원인들 → Top Event로의 인과 경로
- Fault Tree: 이 일이 일어나는 경로는?

**중앙**: 핵심 사건. 예: "데이터센터 정전"

**우측**: Top Event → 결과의 인과 경로
- Event Tree: 이것이 일어났을 때의 영향은?

**장점**: 
- 시각적 명확성. 비기술자도 이해 용이
- 원인과 결과를 동시에 고려
- 통제점(Barriers) 명시

### 예시: 회원 데이터 유출

```
좌측 (원인 → 데이터 유출):
  - 데이터베이스 접근 권한 과다 → (Barrier: 권한 관리) → 유출 방지
  - SQL Injection 취약점 → (Barrier: 코드 검토) → 유출 방지
  
우측 (데이터 유출 → 결과):
  - 언론 보도 → (Barrier: PR 대응) → 평판 피해 제한
  - 고객 이탈 → (Barrier: 신뢰 회복 전략) → 매출 피해 제한
```

### 비즈니스 적용

- 위험도가 높은 프로세스 (금융, 의료, 보안)
- 이해관계자 커뮤니케이션: 경영진, 규제기관 설득

## 시나리오 계획 (Scenario Planning)

### 핵심

**NOT "예측"** (prediction). **"스토리"** (stories).

목표: 미래를 정확히 맞추는 것이 아니라, "여러 가능한 미래"를 준비하는 것.

### 프로세스

**Step 1: 핵심 불확실성 식별**
- "우리의 미래를 결정할 가장 중요한, 예측 불가능한 요소는?"
- 예: AI 기술 발전 속도, 규제 방향, 경쟁자 혁신

**Step 2: 시나리오 구성**
- 보통 2×2 매트릭스로 4가지 시나리오 개발
- 각 시나리오는 **내적으로 일관성 있는 미래상**

예:
```
         高기술발전                低기술발전
高규제   → "규제와 혁신"           → "구조 고착"
LOW규제  → "자유로운 성장"         → "경쟁 정체"
```

**Step 3: 각 시나리오별 전개**
- 이 미래가 왔을 때 우리의 사업은?
- 각 시나리오에서의 기회와 위협은?
- 우리는 어떻게 대응해야 하나?

**Step 4: No-regret Move와 Option 파악**
- **No-regret move**: 모든 시나리오에서 유리한 조치
- **Contingent option**: "이 신호가 보이면 저 방향으로" (적응형)

### 효과

- 경직된 장기 계획보다 유연함
- 팀의 사고 범위 확대 (미래가 현재의 선형 연장이 아님을 인식)
- 이해관계자 간 공통 인식 형성

### 비즈니스 적용

- 장기 전략 수립 (3~5년)
- 신규 시장 진출 결정
- 포트폴리오 다각화 전략

## Red Teaming (적군 역할)

### 정의

상대방의 입장에서 "우리를 어떻게 공격할 것인가"를 생각해보는 활동.

### 프로세스

**Step 1: Red Team 구성**
- 조직과 다른 관점의 사람들
- "새로운 눈으로 보는 사람들" (신입, 외부 전문가, 반대 직군)

**Step 2: 역할 설정**
- Red Team: "상대방" 역 (경쟁자, 규제기관, 해킹팀)
- Blue Team: 우리 입장 (방어)

**Step 3: 공격 시나리오 전개**
- Red Team이 우리의 약점을 찾아 활용하는 방법 제시
- 다단계 공격 경로 (한 번에 뚫리지는 않지만, 여러 단계 조합)

**Step 4: 방어 강화**
- Red Team의 제안에 대해 Blue Team이 대응

### 예시

**Red Team 역할** (금융회사):
1. 구제직원의 계정 탈취 → 
2. 내부 시스템 접근 → 
3. 거래 기록 조작 → 
4. 감시 우회

**Blue Team 대응**:
- 다중인증 강화
- 거래 이상 탐지 AI
- 감시 로그 중앙화 (우회 불가)

### 비즈니스 적용

- 보안 점검 (사이버, 신체)
- 사업 전략 검증 (경쟁자 대응)
- 신규 상품 출시 (시장 반발)

## Reference Class Forecasting

### 배경

Bent Flyvbjerg의 연구. 프로젝트 원가·일정 추정의 체계적 편향 규명.

### 데이터

**258개 프로젝트의 실제 비용초과**:
- 철도 프로젝트: 평균 44.7% 초과
- 다리: 33.8% 초과
- 도로: 20.4% 초과

**통상적 추정**: 
- "우리 프로젝트는 특수하다" → 예상보다 낮게 추정
- 결과: 거의 모든 프로젝트가 초과

### 방법론

**Outside View vs Inside View**

| Inside View | Outside View |
|---|---|
| "이 프로젝트의 세부" | "유사 프로젝트의 실적" |
| 베이스라인 추정 | 기저율 (Reference Class) |
| 위험 요소 가산 (부족함) | 과거 실적 분포 |
| 통상 낙관적 | 통상 정확함 |

### 프로세스

**Step 1: Reference Class 선택**
- "우리 프로젝트와 유사한 과거 프로젝트는?"
- 예: "같은 회사의 크기 비슷한 시스템 통합"

**Step 2: 분포 파악**
- 과거 프로젝트의 실제 원가·일정 데이터 수집
- 평균, 분산, 극값 파악
- "처음 계획 대비 평균 몇% 초과했는가?"

**Step 3: 위치 추정**
- "우리 프로젝트가 이 분포에서 어디에 위치할 것인가?"
- 낙관적? 평균적? 비관적?

**Step 4: 최종 추정**
- Reference Class 평균 + 우리 프로젝트의 특수성
- 통상: Reference Class 평균에 20% 정도 상향

### 효과

- 정확도 40~50% 향상
- 편향 체계적 교정

### 비즈니스 적용

- 프로젝트 착수 전 일정·예산 추정
- 투자 의사결정 전 수익성 검토
- 임원진과의 약속 설정 (현실적 목표)

## 통합 식별 프로토콜

### 상황별 기법 선택

| 상황 | 추천 기법 | 이유 |
|------|----------|------|
| 신규 프로젝트 | Pre-mortem + Reference Class | 초기 위험 식별 + 현실적 추정 |
| 제조/하드웨어 | FMEA | 부품 단위 고장 분석 필수 |
| 전략 의사결정 | Scenario Planning | 미래 불확실성 多 |
| 보안/규정 | Red Teaming | 지능형 위협 고려 |
| 조직 프로세스 | Bow-tie | 원인과 결과 동시 분석 |

### 통합 워크플로우

```
①Pre-mortem으로 초기 리스크 식별
  ↓
②Reference Class Forecasting으로 원가/일정 편향 교정
  ↓
③FMEA 또는 Bow-tie로 상세 분석 (중요 리스크만)
  ↓
④Red Teaming으로 비상식적 공격 경로 점검
  ↓
⑤Scenario Planning으로 대응의 유연성 확보
```
