# 조직·시스템 리스크 이론

## Reason의 Swiss Cheese 모델

### 배경 및 의미

항공 안전, 의료 오류 연구의 거장 James Reason이 개발. 조직의 "인간 오류"를 개인이 아닌 **시스템 설계의 실패**로 본다.

### 모델 구조

**4계층의 방어선**

```
Layer 1: 조직적 영향 (Organizational Influences)
  └─ 부실한 경영진 결정, 예산 부족, 문화적 압박
  └─ 예: "일정을 맞춰야 하니 QA 스킵하자" (경영 지시)

Layer 2: 불안전한 감시 (Unsafe Supervision)
  └─ 형식적 관리, 감시 부재, 부정적 피드백 무시
  └─ 예: "코드 리뷰는 시간 낭비. 그냥 배포하자"

Layer 3: 불안전한 선행조건 (Preconditions for Unsafe Acts)
  └─ 피로, 스트레스, 불완전한 정보, 도구 부족
  └─ 예: "개발자가 3일 밤샘 코딩 후 배포"

Layer 4: 불안전한 행동 (Unsafe Acts)
  └─ 개별 실수, 위반, 약한 상황판단
  └─ 예: "배포 중 잘못된 환경변수 입력"
```

### 핵심: 치즈 구멍

각 계층이 "치즈"고, "구멍"이 있다. 구멍이 우연히 정렬되면 사고 발생.

**전통적 대응 (❌)**
- Layer 4의 개인만 책임: "개발자가 실수했다 → 징계/재교육"
- 결과: 문제 반복 (같은 상황이 재발하면 누군가 또 실수함)

**올바른 대응 (✓)**
- Layer 1~3의 **잠재적 실패** (Latent failures) 제거
- 예: 충분한 QA 시간 할당 → Layer 3 개선
- 예: 경영진이 "일정 vs 품질" 신호 명확히 → Layer 1 개선

### 적극적 vs 잠재적 실패

| | 활동 오류 (Active Failures) | 잠재적 오류 (Latent Failures) |
|---|---|---|
| 시점 | 즉각적 | 오래전 |
| 가시성 | 눈에 띔 | 숨어있음 |
| 원인 | "어떻게 그렇게 실수?" | "왜 그런 상황을 만들었나?" |
| 책임 | 개인 | 시스템·조직 |
| 방지 | 개인 훈련 (제한적) | **시스템 설계 개선** |

**비즈니스 의미**: 사고·오류 발생 시 개인을 찾기 전에 "어떤 시스템 결함이 이를 가능하게 했는가?"를 먼저 묻기.

## Perrow의 정상사고 이론 (Normal Accident Theory)

### 배경

시스템이 어느 정도 규모에 이르면, 설계가 아무리 완벽해도 사고는 **불가피하다**. Three Mile Island, Deepwater Horizon 등 분석.

### 핵심 메커니즘: 2가지 차원

#### 차원 1: 상호작용 복잡도 (Interactive Complexity)

**정의**: 시스템의 부분들이 예상 밖의 방식으로 상호작용하는 정도

| 낮은 복잡도 | 높은 복잡도 |
|---|---|
| 선형 프로세스 (A→B→C) | 순환 피드백 (A→B→C→A) |
| 한 사람이 전체를 이해 가능 | 어떤 한 사람도 전체 영향을 예측 불가 |
| 고장 원인 추적 용이 | 고장 원인이 다층적·연쇄적 |
| 예: 자동차 조립 | 예: 원전, 금융 시스템 |

#### 차원 2: 결합도 (Coupling)

**정의**: 시스템 부분들 간의 시간적·공간적 근접성

| 느슨한 결합 | 타이트 결합 |
|---|---|
| 한 부분 실패 → 다른 부분이 버퀨려함 | 한 부분 실패 → 즉시 연쇄 영향 |
| 중간에 완충 시간/자원 가능 | 지연 없이 실패 전파 |
| 예: 대학 (학부 독립적) | 예: 원전 (실시간 제어 필수) |

### 정상사고 매트릭스

```
                낮은 복잡도              높은 복잡도
느슨한 결합      선형·통제 가능           학습·적응 필요
                (사고 드뭄)              (사고 예상하되 완화 가능)

타이트 결합      예측 가능한 에러          정상사고 불가피
                (신호 감시로 관리)        (설계부터 다시)
```

**정상사고 영역** (우하단): 높은 복잡도 + 타이트 결합
- 예: 원전 (원자로 상태 예측 불가, 냉각 지연 불가)
- 예: Deepwater Horizon (석유 시추 시스템, 시간당 제어 불가)
- 예: 복잡한 금융 파생상품 (상관관계 비선형, 실시간 대응 필수)

### 임의성 (Incompleteness of Design)

**핵심 통찰**: 아무리 충분한 설계도, 반드시 "빈틈"이 있다. 왜?
1. 설계자가 상상하지 못한 상황 존재 (unknown unknowns)
2. 설계 문서와 실제 운영이 다름 (부정행위, 편의, 변형)
3. 여러 결함이 동시 발생할 확률은 0이 아님

### 의미와 대응

**전통적 대응 (❌)**
- "더 많은 교육, 감시, 프로세스"
- 결과: 시스템이 더 복잡해짐 + 사고 여전히 발생

**올바른 대응 (✓)**
- 시스템 설계 자체를 변경
- 느슨한 결합으로 변경: 모듈화, 의존성 감소, 버퍼 추가
- 복잡도 감소: 예측 가능한 부분과 비예측 부분 분리

**비즈니스 예시**
- 스타트업: "모든 게 연결되어 있다" (타이트 결합) + "누가 뭘 하는지 모른다" (높은 복잡도)
  → 한 명의 이탈 = 전체 마비. 이를 설계로 해결해야 함
- 마이크로서비스: 각 서비스가 독립적 (느슨한 결합) → 정상사고 확률 감소

## Weick의 고신뢰도 조직 (HRO: High-Reliability Organizations)

### 배경

핵심 질문: "어떤 조직은 극도로 복잡한 환경에서도 안전하게 운영되는가?"
- 핵심: 항공모함, 원전 통제실, 응급실
- 공통점: 실패의 여지가 없음 + 실패 확률은 0이 아님 → 어떻게?

### 5가지 원칙

#### 1. 실패에 대한 집착 (Preoccupation with Failure)

**특징**:
- "우리는 안전하다"는 생각 금지
- 작은 incident를 canary로 봄
- "왜 문제가 없는가?"를 반문
- 근본 원인 분석을 조직 문화로

**비즈니스 의미**
- Near-miss 추적: 실제 사고가 아니었지만 운이 좋았던 경우
- 예: "배포 중 잘못된 환경변수 입력했는데 운 좋게 감지됨" → 이것이 canary
- 사고 보고 문화: "누가 실수했는가?" X → "어떤 시스템 결함이 있었나?" O

#### 2. 단순화 저항 (Reluctance to Simplify)

**특징**:
- 복잡하고 모순적인 정보를 받아들임
- "일반화하기"를 경계
- 라벨·카테고리를 거부 ("그게 다가 아니야")
- 정보의 미묘한 차이 유지

**비즈니스 의미**
- Risk 스캔 후 "이건 Green"이라고 마무리하면 ✗
- "이건 일반적으로는 Green인데, 이 특수 상황에서는 Orange 가능성" ✓
- 조직 내 "다양한 관점"을 유지 (한 가지 내러티브 금지)

#### 3. 운영에 대한 민감성 (Sensitivity to Operations)

**특징**:
- 실시간 현장 데이터 중시
- 보고서·요약 X, 직접 관찰·청취
- "큰 그림"보다 "작은 신호" 포착
- 경영진이 현장과 접촉 유지

**비즈니스 의미**
- CEO가 월간 요약만 받으면 ✗
- CEO가 주간 스탠드업, 고객 피드백을 직접 들으면 ✓
- 대시보드는 도구일 뿐, 인간적 대화가 핵심

#### 4. 회복력에 대한 헌신 (Commitment to Resilience)

**특징**:
- 사고 발생 시 빠른 대응·복구
- "사고를 완전히 예방"이 아니라 "사고에서 빠르게 회복"을 목표
- 훈련, 시뮬레이션, 대응 계획
- 사고 후 학습과 적응

**비즈니스 의미**
- 장애 발생 시 "범인 찾기" X → "1시간 안에 복구하고 원인 분석" ✓
- Disaster recovery plan + 정기 훈련
- "100% 가동시간" 목표 X → "사고 발생 시 2시간 내 복구" 목표 ✓

#### 5. 전문성에 대한 존경 (Deference to Expertise)

**특징**:
- 권한은 직급이 아니라 전문성에 따라 흐름
- 현장 전문가의 판단을 존중
- 하급자가 상급자의 의견을 이의제기 가능
- "실장님이 말씀하셨으니" X, "그 부분 가장 잘 아는 사람이 결정" ✓

**비즈니스 의미**
- 경영진의 의견이 항상 맞지는 않음
- 엔지니어가 "이건 위험하다"고 말하면 경청 필수
- 의사결정 권한을 지식에 맞춰 분산

### HRO 적용 체크리스트

| 원칙 | 조직의 현황 체크 |
|------|---|
| 실패 집착 | Near-miss를 추적하고 공유하는 문화가 있는가? |
| 단순화 저항 | 한 가지 내러티브에 중독되지 않는가? |
| 운영 민감성 | 경영진이 현장과 정기적으로 접촉하는가? |
| 회복력 | 사고 후 학습과 개선이 체계화되어 있는가? |
| 전문성 존경 | 의사결정이 직급이 아니라 전문성 중심인가? |

## Kaplan & Mikes의 3가지 리스크 카테고리

### 개요

모든 리스크를 동일하게 관리하면 실패한다. 리스크 타입에 따라 관리 구조·프로세스·의사결정이 달라야 한다.

### 3가지 카테고리와 관리 방식

#### 1. 예방가능 리스크 (Preventable Risks)

**정의**: 조직이 의도적으로 통제할 수 있는 리스크. 비즈니스 운영상 "발생하면 안 되는" 것.

**예시**
- 규정 위반, 부정행위, 내부 통제 실패
- 안전 사고, 환경오염, 감염병
- 데이터 유출, IT 장애

**관리 방식 (규칙 기반)**
- 명확한 규칙과 프로세스 수립
- 모니터링과 감시 강화
- 위반 시 즉각적 징계
- 목표: 발생 건수를 0에 가깝게

**조직 담당**: 운영 팀, Compliance, Risk Management

**경영진 역할**: 규칙 설정 후 "실행"에 의존

#### 2. 전략 리스크 (Strategy Risks)

**정의**: 경영진의 전략적 선택의 결과로 발생하는 리스크. "무엇을 할 것인가"의 선택에서 나옴.

**예시**
- 신규 시장 진출 실패
- 인수합병 가치 창출 실패
- 신기술 투자 수익화 실패
- 조직 구조 개편의 부작용

**관리 방식 (대화 기반)**
- 예/아니오 답이 없음. 다양한 관점 청취
- 가정을 명시하고 검증
- 빠른 학습과 적응 (방향은 정했되, 세부는 가면서 조정)
- 실패로부터 배우고 비용을 빠르게 제한

**조직 담당**: CEO, 이사회, 경영진 토론

**의사결정**: 분석보다 "직관+경험+다양한 의견"

#### 3. 외부 리스크 (External Risks)

**정의**: 조직이 통제할 수 없는 외부 환경의 불확실성

**예시**
- 경제 침체, 금리 변화
- 경쟁자의 예기치 않은 행동
- 자연재해, 팬데믹
- 규제 변화
- 지정학적 위기

**관리 방식 (시나리오 기반)**
- 여러 시나리오 개발 (낙관/기본/비관)
- 각 시나리오별 대응 계획
- 금융적 헤징, 보험
- 자산 배치의 유연성 (옵션 보유)

**조직 담당**: 전체 조직 (모두에게 영향)

**의사결정**: 확률을 정할 수 없으므로, "가장 나쁜 경우를 어디까지 견딜 수 있는가?"

### 관리 매트릭스

| | 예방가능 | 전략 | 외부 |
|---|---|---|---|
| **의사결정 형태** | 규칙 기반 | 대화 기반 | 시나리오 기반 |
| **조직 구조** | 계층적. 명확한 책임 | 대각선. 토론 문화 | 통합적. 전사 영향 |
| **핵심 지표** | 발생 건수 (목표: 0) | 전략 가정 검증 → 적응 속도 | 자산 탄력성, 회복력 |
| **실패의 의미** | 프로세스 실패 → 재점검 필수 | 학습 기회 → 가정 수정 | 시나리오 재평가 필수 |
| **시간 범위** | 단기 (분기) | 중기 (1~2년) | 장기 (다년도) |

## COSO ERM & ISO 31000

### 통합 프레임

현대적 리스크 관리는 위의 이론들(Reason, Perrow, Weick, K&M)을 통합하는 구조를 제공:

**COSO 5가지 컴포넌트**
1. 거버넌스·문화 (Reason의 조직 영향)
2. 전략·목표 설정 (K&M의 전략 리스크)
3. 리스크 식별·분석 (Perrow의 시스템 분류)
4. 대응 설계 (Weick의 회복력, HRO 원칙)
5. 모니터링 (Weick의 운영 민감성)

**ISO 31000 원칙**
- Integrated (모든 수준에서)
- Structured (일관된 프로세스)
- Customized (조직 맥락 반영)
- Inclusive (모든 이해관계자)
- Dynamic (환경 변화 감지)
- Transparent (명확한 커뮤니케이션)
- Informed (데이터 기반)
- Accountable (책임 명확)
